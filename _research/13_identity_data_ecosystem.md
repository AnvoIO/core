# Identity & Data Ecosystem

## Vision

The chain isn't just an L1 — it's the foundation for a **sovereign identity and
data platform**. Every person owns their identity. Every person controls their data.
Applications consume identity and data with the user's permission, not the platform's.

This inverts the current internet model where platforms own your identity (Google login),
your data (Facebook's social graph), and your reputation (Uber's rating). On this chain,
those belong to you.

## The Stack

```
Layer 4: Applications
         Social, governance, finance, marketplaces, games
         (consume identity + data with user permission)
              |
Layer 3: Data Ecosystem
         User-controlled data vaults, selective disclosure,
         verifiable credentials, data marketplace
              |
Layer 2: Identity
         Proof of personhood, attestations, reputation,
         social graph, credential anchoring
              |
Layer 1: Chain
         Passkey accounts, permissions, free baseline,
         gas model, parallel execution, BTC/ETH bridges
```

Each layer enables the next. The chain gives you accounts and permissions.
Identity proves accounts are real people. Data gives those people ownership
of their information. Applications consume all of it with user consent.

## Layer 2: Identity

### Proof of Personhood — Layered, Not Binary

Personhood isn't a single check — it's a **confidence spectrum**. Multiple
independent signals stack to increase confidence that an account represents
a unique real person.

#### Signal Layers

| Layer | Signal | Privacy | Sybil Resistance | How It Works |
|---|---|---|---|---|
| 0 | **Account exists** | Full | None | Has a passkey, passed deposit |
| 1 | **Hardware attestation** | Full | Medium | Passkey attestation proves real hardware device |
| 2 | **Social vouching** | Full | Medium-High | N verified users vouch for this person |
| 3 | **Biometric uniqueness (ZK)** | Full (ZK proof) | Very high | Prove you're unique without revealing biometrics |
| 4 | **Credential attestation** | Configurable | High | Third-party attestor (KYC provider, gov ID, university) |
| 5 | **Reputation accrual** | Full | Emergent | Transaction history, community standing, time on chain |

Each layer is independent and additive. A user might have layers 0+1+2 (hardware +
social vouching) without ever doing KYC. Another user might have layers 0+4 (account +
KYC attestation) for a regulated DeFi application. Applications declare which layers
they require.

#### Layer 0: Account Exists (Already Built)

Passkey account with refundable deposit. Account has baseline allocation.
This is the floor — proves someone spent tokens to create the account.

#### Layer 1: Hardware Attestation (Passkey-Based)

WebAuthn attestation certificates prove the passkey lives on a real hardware
device (iPhone Secure Enclave, Android Titan chip, YubiKey, etc.).

```
During account creation:
  → navigator.credentials.create() returns attestation object
  → attestation contains manufacturer certificate chain
  → verifier confirms: "this key was generated by a real Apple/Google/Yubico device"
  → on-chain: attestation hash stored as identity signal
```

**Privacy:** No personal data revealed. Just "this is a real hardware device."

**Sybil resistance:** Medium — a person can own multiple devices, but each
device is a real physical object with a cost.

**Implementation:** Contract stores attestation metadata. Verification can
happen off-chain (attestation cert chains are complex) with the result
anchored on-chain as an attestation.

#### Layer 2: Social Vouching (Web of Trust)

Existing verified users vouch for new users. Creates a social graph where
sybil accounts are expensive to create (need real people to vouch).

```
Identity contract:
  vouch(voucher: account, vouchee: account)
    → voucher must be verified at layer 2+ themselves
    → each account can vouch for N people (e.g., 5)
    → vouches replenish slowly (1 per month)
    → vouchee becomes layer 2 verified after M vouches (e.g., 3)

Anti-gaming:
  → if vouchee is later flagged as sybil, voucher loses vouching power
  → creates accountability in the social graph
  → vouching carelessly has consequences
```

**Privacy:** Full — no personal data. Just social links (which can be
privacy-shielded if needed).

**Sybil resistance:** Medium-High — creating fake accounts requires
convincing real verified users to vouch. Gaming the social graph is
expensive and risky (vouchers penalized for bad vouches).

#### Layer 3: Biometric Uniqueness (ZK Proofs)

The strongest privacy-preserving proof of personhood. Prove you're a unique
human without revealing any biometric data.

```
Enrollment:
  → user scans face/fingerprint via device
  → device computes biometric hash locally (never leaves device)
  → ZK circuit proves: "I have a biometric that hasn't been enrolled before"
  → proof submitted on-chain, verified via alt_bn128 pairing intrinsics
  → on-chain: nullifier stored (prevents re-enrollment), no biometric data

Verification:
  → any contract can check: does this account have a valid uniqueness proof?
  → no biometric data on-chain, no biometric data shared with anyone
```

**Privacy:** Maximum — zero-knowledge proof reveals nothing about the person.

**Sybil resistance:** Very high — one body, one proof. Can't create duplicates
without a different biometric.

**Technical foundation:** Spring already has `alt_bn128_add/mul/pair` intrinsics
for ZK proof verification. The ZK circuits run off-chain; only the proof is
verified on-chain.

**Approaches:**
- **Worldcoin/World ID style:** iris scan → ZK proof of uniqueness
- **Facial recognition hash:** less invasive, lower confidence
- **Fingerprint commitment:** device-local, privacy-preserving

This is the most technically complex layer. Could integrate with existing
proof-of-personhood protocols (World ID, Humanode, BrightID) rather than
building from scratch.

#### Layer 4: Credential Attestation

Third-party attestors verify claims about the user and anchor the result
on-chain. The attestation says "this claim was verified" without revealing
the underlying data.

```
Attestation framework:
  attestor: "kyc-provider.chain"
  subject: "alice"
  claim: "age_over_18"
  evidence: hash(kyc_document)  // document NOT on-chain
  expiry: 2027-01-01
  signature: attestor signs (subject, claim, evidence, expiry)

On-chain storage:
  attestation table: (subject, attestor, claim, expiry, signature)
  → anyone can verify: "kyc-provider attested alice is over 18"
  → nobody can see the KYC document itself
```

**Attestation types:**
- KYC verification (age, jurisdiction, accredited investor)
- Educational credentials (degrees, certifications)
- Employment verification
- Government ID anchoring (without storing the ID)
- Professional licenses
- Credit attestations

**Who can be an attestor:**
- Registered attestor accounts (governance-approved)
- Anyone can create an attestation, but applications choose which attestors to trust
- Market for attestation services — competition on price, speed, coverage

**Standards alignment:** Compatible with W3C Verifiable Credentials and DID
standards. Attestations can be exported as VCs for use off-chain.

#### Layer 5: Reputation Accrual

Organic reputation built through on-chain activity over time. Not a single
score — a multi-dimensional profile.

```
Reputation dimensions:
  → account_age: time since creation
  → transaction_volume: total transactions executed
  → governance_participation: votes cast, proposals made
  → attestation_count: number of valid attestations held
  → social_standing: vouching graph position
  → economic_activity: staking, bridging, trading history
  → community_contributions: contract deployments, public goods funding
```

No single "reputation score" — applications query the dimensions they care about.
A DeFi protocol might weight economic activity. A governance system might weight
participation history. A social platform might weight social graph position.

**All derived from on-chain data** — no oracle needed. Contracts can read
other contracts' tables to compute reputation inputs.

### Identity Permission Integration

The account permission system (doc 12) natively supports identity-gated actions:

```
Account "alice":
  owner (threshold: 1)
    - passkey [phone, weight: 1]

  active (threshold: 1)
    - passkey [phone, weight: 1]

  verified (threshold: 1)
    - identity_contract@core.code [weight: 1]
    → this permission is satisfied when identity_contract
      confirms alice has layer 2+ verification

  kyc_verified (threshold: 1)
    - kyc_attestation_contract@core.code [weight: 1]
    → satisfied when alice has a valid KYC attestation
```

Applications link their actions to the appropriate permission level:
```
// Social platform: requires verified (layer 2+)
linkauth(alice, social_contract, post, verified)

// DeFi: requires kyc_verified (layer 4)
linkauth(alice, defi_contract, large_trade, kyc_verified)

// Game: no identity required (just active permission)
// (no linkauth needed, defaults to active)
```

This is **already supported** by the existing permission and linkauth system.
No node changes needed.

## Layer 3: Data Ecosystem

### User-Controlled Data Vaults

Each user has a **data vault** — a contract-managed store of personal data
where the user controls access.

```
Data vault for "alice":

  Profile:
    display_name: "Alice"           [public]
    email: "alice@example.com"      [encrypted, shared with: dapp1, dapp2]
    phone: "+1234567890"            [encrypted, shared with: kyc_provider]
    birthdate: "1990-01-15"         [encrypted, not shared]

  Social:
    connections: [bob, carol, dave] [public]
    messages: [...]                 [encrypted, per-conversation keys]

  Financial:
    credit_score_attestation: {...} [encrypted, shared with: defi_protocol]
    income_attestation: {...}       [encrypted, shared with: mortgage_dapp]

  Activity:
    purchase_history: [...]         [encrypted, shared with: recommendation_engine]
    browsing_preferences: [...]     [encrypted, not shared]
```

### Selective Disclosure

Users grant and revoke access to specific data fields for specific applications.
Access is managed through the permission system:

```
Alice grants email access to dapp1:
  → alice calls vault::grant(dapp1, "email")
  → vault re-encrypts email with dapp1's public key
  → dapp1 can read alice's email
  → alice can revoke anytime: vault::revoke(dapp1, "email")
  → on revoke: dapp1's access key is rotated, can no longer decrypt
```

**Encryption model:**
- Data encrypted at rest with user's key
- Shared data re-encrypted with recipient's key (proxy re-encryption or per-field keys)
- On-chain: encrypted blobs + access control table
- Off-chain (optional): large data stored on IPFS/Arweave, hash anchored on-chain

### Verifiable Credentials

The attestation framework (Layer 2, Level 4) produces credentials that can be
used as verifiable claims in the data ecosystem:

```
Alice applies for a loan:
  → DeFi protocol requests: credit_score, income, identity_verification
  → Alice's wallet shows: "LoanDAO wants access to: credit score, income, identity"
  → Alice approves selectively: shares credit + identity, denies income
  → DeFi protocol receives:
      - credit_score attestation (signed by credit_attestor)
      - identity attestation (signed by kyc_provider)
      - income: ACCESS DENIED
  → DeFi protocol decides based on available data
```

### Data Marketplace

Users can **sell** access to their data:

```
Data buyer (e.g., research firm) posts:
  "Buying anonymized purchase history, 0.10 TOKEN per user per month"

Users opt in:
  → alice calls marketplace::sell(buyer, "purchase_history", price)
  → data exported anonymized (ZK proof: "this data comes from a verified account
    with these demographic attestations, but you can't identify who")
  → alice receives 0.10 TOKEN/month
  → alice can opt out anytime

No intermediary:
  → buyer pays alice directly (or via escrow contract)
  → no Facebook/Google taking 100% of the data value
  → users earn from their own data
```

### Portability

Because identity and data live on-chain (or are anchored on-chain), they're
**portable across every application on the chain**:

```
Alice joins a new social platform:
  → "Import your profile from your data vault?"
  → one click: display name, connections, preferences imported
  → no re-entering information
  → no "sign up with Google" (where Google owns the identity)

Alice switches DeFi protocols:
  → credit attestation, trading history, reputation all carry over
  → new protocol can read alice's on-chain reputation dimensions
  → no cold-start problem
```

## What This Enables

### Sybil-Resistant Governance

```
One person = one vote (not one token = one vote)
  → governance_contract requires identity.verified permission
  → permission satisfied only if identity_contract confirms layer 2+ personhood
  → whales can't buy extra votes
  → DAOs with actual democratic governance
```

### Fair Token Distributions

```
Airdrop to verified humans only:
  → airdrop_contract checks identity layer for each claimant
  → one claim per verified person
  → no sybil farming, no airdrop bots
```

### UBI / Faucet Systems

```
Universal basic income on-chain:
  → ubi_contract distributes X tokens per verified person per period
  → layer 3 (biometric uniqueness) prevents double-claiming
  → privacy-preserving: ZK proof, no biometric on-chain
```

### Reputation-Based Lending

```
Under-collateralized loans:
  → lending_contract checks reputation dimensions:
      account_age > 1 year
      transaction_volume > 1000
      no default history
      identity layer 2+ verified
  → loan offered based on on-chain reputation, not off-chain credit score
  → or: attestation from credit bureau anchored on-chain (layer 4)
```

### Social Platforms Without Platform Risk

```
Social network where:
  → your profile is your data vault (you own it)
  → your social graph is on-chain (portable)
  → your content is signed by your passkey (authentic)
  → you can move to a competing frontend without losing anything
  → no deplatforming (your data is yours, not the platform's)
  → identity-verified posting (optional, reduces trolling/bots)
```

### Quadratic Funding

```
Public goods funding:
  → quadratic_funding_contract weights contributions by √(amount)
  → but ONLY from verified unique humans (layer 3)
  → prevents sybil attacks that game quadratic formulas
  → Gitcoin-style funding with real sybil resistance
```

## Antelope Chain Findings: Identity

### UX Network — eosio.info On-Chain KYC Framework

UX Network ships a production on-chain identity layer (`eosio.info`) with a clean
multi-provider design. Four tables:

| Table | Scope | Purpose |
|---|---|---|
| `kycaccounts` | global | Whitelisted KYC provider accounts (managed by `eosio`) |
| `keytypes` | global | Definitions of verification and info key types, with `user` boolean distinguishing user-provided vs provider-verified |
| `userverifs` | per provider | Verification records added by KYC providers. Composite key index `(user, verification_key)` with secondary indices by user and by key |
| `userkeys` | per user | User-controlled info records (self-declared data) |

Key design patterns:
- **Multiple independent providers** — any whitelisted KYC provider can add verifications
- **User sovereignty** — users control their own `userkeys` data independently
- **Composite key indexing** — `(uint128_t{x} << 64) | y` for efficient cross-key lookups
- **Separation of user data and verified data** — the `user` boolean on key types distinguishes
  self-declared information from third-party verifications

**Discussion:** This maps well to Anvo's layered identity model:
- `kycaccounts` ≈ Layer 4 attestor registry
- `userverifs` ≈ Layer 4 credential attestations
- `userkeys` ≈ User-controlled data vault (Layer 3/Data Ecosystem)
- The composite key pattern is efficient and should be adopted

UX's design is simpler than Anvo's full vision (no social vouching, no ZK biometrics,
no reputation) but provides a solid structural foundation for the attestation
framework. The multi-provider model is especially important — avoids single-provider
lock-in and allows different providers for different jurisdictions/use cases.

**Recommendation:** Use UX's table structure as a starting template for Anvo's
attestation contract, then extend with:
- Attestation expiry timestamps
- Revocation support
- Layer classification (which identity layer does this attestation satisfy?)
- Privacy flags (on-chain hash vs. full data)

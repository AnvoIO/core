# Testing Guide

This document covers the testing infrastructure, how to run tests, and how to write new ones.

---

## Test Frameworks

This repo has two testing layers:

| Framework | Location | Language | Purpose |
|-----------|----------|----------|---------|
| **Unit tests** | `unittests/`, `plugins/*/tests/` | C++ (Boost.Test) | Individual component tests — crypto, chain logic, serialization, protocol types |
| **Integration / E2E tests** | `tests/` | Python (TestHarness) | Multi-node cluster tests — P2P peering, sync, producer handoff, consensus transitions |

**anvo-test** (`@anvoio/anvo-test` npm package) is a separate framework for smart contract testing. It runs a single Docker node and is not used for core node testing.

---

## Running Tests

### Build

```bash
cd build
cmake .. -GNinja -DCMAKE_CXX_SCAN_FOR_MODULES=OFF
ninja
```

### Unit tests

```bash
# All parallelizable tests
ctest --output-on-failure -j $(nproc) -LE "nonparallelizable_tests|long_running_tests" --timeout 480

# Specific test binary
./plugins/net_plugin/tests/test_net_plugin
./unittests/unit_test --run_test=block_header_state_tests
./libraries/libfc/test/test_fc
```

### Integration tests (non-parallelizable)

```bash
# All non-parallelizable tests (run sequentially)
ctest --output-on-failure -j 1 -L nonparallelizable_tests --timeout 480

# Single test
ctest -R p2p_encrypted_transport_test --output-on-failure
```

### Long-running tests

```bash
ctest --output-on-failure -j 1 -L long_running_tests --timeout 1800
```

## Unit Test Structure

Unit tests live alongside the code they test:

```
plugins/net_plugin/tests/
  CMakeLists.txt                          # test target definition
  main.cpp                               # Boost.Test module entry
  auto_bp_peering_unittest.cpp           # BP auto-peering logic
  rate_limit_parse_unittest.cpp          # rate limit config parsing
  p2p_transport_encryption_unittest.cpp  # ECDH, AEAD, HKDF, node key, protocol types
```

Each test file uses Boost.Test:

```cpp
#include <boost/test/unit_test.hpp>

BOOST_AUTO_TEST_SUITE(my_tests)

BOOST_AUTO_TEST_CASE(my_test_name) {
    // test body
}

BOOST_AUTO_TEST_SUITE_END()
```

To add a new test file, add it to the `CMakeLists.txt` in the test directory:

```cmake
add_executable( test_net_plugin
    existing_test.cpp
    my_new_test.cpp        # add here
    main.cpp
)
```

## Integration Test Infrastructure (TestHarness)

Multi-node integration tests use the Python TestHarness at `tests/TestHarness/`.

### Key components

| File | Purpose |
|------|---------|
| `Cluster.py` | Launches and manages a cluster of `core_netd` nodes |
| `Node.py` | Controls a single `core_netd` process — start, stop, RPC queries |
| `launcher.py` | Generates network topology and P2P peering configuration |
| `TestHelper.py` | Argument parser and test lifecycle (setup/teardown) |
| `WalletMgr.py` | Manages keosd wallet for key management |
| `queries.py` | HTTP RPC wrappers for `/v1/chain/*`, `/v1/net/*` endpoints |
| `testUtils.py` | Utility functions — port checking, process management |

### How multi-node tests work

1. **`Cluster.launch()`** spawns N `core_netd` processes on localhost with unique ports
2. **Topology** is configured via built-in generators or custom JSON shape files:
   - `mesh` — all nodes connected to all nodes
   - `line` — linear chain (A→B→C)
   - `star` — hub-and-spoke
   - `ring` — line with endpoints connected
   - Custom JSON — define exact peer relationships
3. **P2P peering** is configured via `--p2p-peer-address` flags generated by `launcher.py`
4. **Tests** interact with nodes via HTTP RPC and verify sync state
5. **`Cluster.shutdown()`** kills all processes and cleans up

### Writing a new multi-node test

```python
#!/usr/bin/env python3

import signal
from TestHarness import Cluster, TestHelper, Utils

###############################################################
# my_p2p_test
#
# Test description here.
###############################################################

Print = Utils.Print

def main():
    args = TestHelper.parse_args({
        "-p": 2,        # producer nodes
        "-n": 3,        # total nodes
        "-d": 1,        # delay between nodes
        "-s": "mesh",   # topology
    })

    cluster = Cluster(loggingLevel="all", keepRunning=False, keepLogs=False)
    cluster.setWalletMgr(TestHelper.getWalletMgr())

    try:
        TestHelper.printSystemInfo("BEGIN")

        # Per-node custom args
        specificArgs = {
            0: "--p2p-require-encryption true",
            1: "--p2p-require-encryption false",
        }

        cluster.launch(
            pnodes=args.p,
            totalNodes=args.n,
            delay=args.d,
            topo=args.s,
            specificExtraNodeosArgs=specificArgs,
        )

        # Wait for cluster to sync
        assert cluster.waitOnClusterSync(blockAdvancing=5), "Cluster failed to sync"

        # Access individual nodes
        node0 = cluster.getNode(0)
        node1 = cluster.getNode(1)

        # Query node state via RPC
        info = node0.getInfo()
        Print(f"Node 0 head block: {info['head_block_num']}")

        # Check P2P connections via net_api
        connections = node0.processUrllibRequest("net", "connections")
        Print(f"Node 0 connections: {len(connections['payload'])}")

        # Wait for LIB to advance (proves consensus works)
        assert node0.waitForLibToAdvance(), "LIB did not advance"

        TestHelper.printSystemInfo("END")
        errorCode = 0

    except Exception as e:
        Print(f"Test failed: {e}")
        errorCode = 1
    finally:
        TestHelper.shutdown(cluster, None, not args.keep_logs, errorCode != 0)

    exit(errorCode)

if __name__ == "__main__":
    main()
```

### Registering in CI

Add to `tests/CMakeLists.txt`:

```cmake
# Non-parallelizable (multi-node, uses shared ports)
add_np_test(NAME my_p2p_test COMMAND tests/my_p2p_test.py)

# Long-running (>8 min timeout)
add_lr_test(NAME my_long_test COMMAND tests/my_long_test.py)
```

### Custom topology via JSON shape files

Shape files define exact node relationships:

```json
{
  "name": "testnet_",
  "nodes": {
    "bios": {
      "keys": [{"pubkey": "PUB_K1_...", "privkey": "PVT_K1_..."}],
      "peers": [],
      "producers": ["eosio"]
    },
    "testnet_00": {
      "keys": [{"pubkey": "PUB_K1_...", "privkey": "PVT_K1_..."}],
      "peers": ["bios", "testnet_01"],
      "producers": ["defproducera"]
    },
    "testnet_01": {
      "peers": ["bios", "testnet_00"],
      "producers": ["defproducerb"]
    }
  }
}
```

Pass with: `--cluster-topology-type path/to/shape.json`

### Per-node custom arguments

Use `specificExtraNodeosArgs` dict to pass different flags to different nodes:

```python
specificArgs = {
    0: "--p2p-require-encryption true --agent-name node0",
    1: "--p2p-max-total-buffer-bytes 104857600",
    2: "",  # default args
}
cluster.launch(..., specificExtraNodeosArgs=specificArgs)
```

### Inspecting P2P state

```python
# Get active P2P connections
connections = node.processUrllibRequest("net", "connections")

# Check node logs for P2P events
# Logs are written to: var/lib/node_NN/stderr.txt
```

## Antelope / Spring Compatibility Testing

This is a manual validation step, not part of CI. It verifies that nodes built from this repo can peer with unmodified Antelope/Spring nodes.

### Setup

1. Download the latest Spring release binary:
   ```bash
   # From https://github.com/AntelopeIO/spring/releases
   wget https://github.com/AntelopeIO/spring/releases/download/v1.2.2/spring_1.2.2-ubuntu22.04_amd64.deb
   sudo dpkg -i spring_1.2.2-ubuntu22.04_amd64.deb
   # Binary at /usr/bin/nodeos
   ```

2. Launch a mixed cluster using the TestHarness with `--alternate-version-labels-file`:
   ```bash
   # The TestHarness supports launching specific nodes with different binaries
   # via the alternateVersionLabels mechanism in launcher.py
   ```

3. Or manually:
   ```bash
   # Terminal 1: Spring V1 node
   nodeos --p2p-listen-endpoint 0.0.0.0:9876 --data-dir ./v1-data --config-dir ./v1-config

   # Terminal 2: Anvo V2 node
   core_netd --p2p-listen-endpoint 0.0.0.0:9877 --p2p-peer-address 127.0.0.1:9876 \
             --data-dir ./v2-data --config-dir ./v2-config
   ```

### What to verify

| Scenario | Expected |
|----------|----------|
| V2 node connects to V1 node | Plaintext connection, sync works |
| V1 node connects to V2 node | Plaintext connection, sync works |
| V2 node with `--p2p-require-encryption` connects to V1 node | Connection rejected (go_away: authentication) |
| V2 node connects to V2 node | Encrypted connection (ChaCha20-Poly1305) |
| Mixed cluster produces blocks | All nodes receive blocks, LIB advances |

### When to run

Run this validation when:
- Bumping `network_version` / `proto_version_t`
- Adding new message types to `net_message` variant
- Changing handshake processing or version negotiation logic
- Before any release that modifies the P2P layer

## Test Categories in CI

| Category | Label | Timeout | Parallelism | Examples |
|----------|-------|---------|-------------|---------|
| Parallelizable | (default) | 480s | `nproc` | `test_net_plugin`, `abi_tests`, `wallet_tests` |
| Non-parallelizable | `nonparallelizable_tests` | 480s | 1 (sharded) | Multi-node P2P tests, production restart |
| Long-running | `long_running_tests` | 1800s | 1 | Extended consensus tests, large sync tests |
